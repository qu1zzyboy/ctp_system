# CTP 多账户交易服务 — 机考提交说明

> 项目：`ctp_system`
> 对应题目：[机试题目-基于CTP接口的多账户交易服务](../机试题目-基于CTP接口的多账户交易服务.md)
> 更新日期：2026-07-14

---

## 1. 项目说明

本项目基于 CTP Trade 与 MD 接口，实现一个多 Client、单 Server 的 C/S 架构交易服务系统。Server 统一管理 CTP 行情连接、交易账户连接、Client 权限、交易指令路由和回报推送；Client 作为交易终端向 Server 发送登录、查询、下单、撤单、登出和行情订阅指令。

核心接入模式为：

- 1 个共享 CTP MD 行情连接。
- N 个 CTP Trade 交易连接，每个 CTP 账户一个 `TradeSessionActor`。
- 多个 Client 可同时连接同一个 Server。
- Server 端通过 TOML 白名单控制 Client 对账户的读写权限。

## 3. 架构说明

系统按 Client、Internal Gateway、Server Actor、Exchange Adapter、CTP 前置五层组织。

Server 启动时只创建两个内部 TCP 监听和共享状态：

- Trading TCP Server：监听 `0.0.0.0:8000`，处理 `Hello`、账户登录、下单、撤单、查询和登出。
- MarketData TCP Server：监听 `0.0.0.0:8001`，处理 `MarketDataHello`、行情订阅和退订。
- `ServerActorState`：维护 Client 连接、Client-Account 白名单、交易账户 Actor、行情订阅表和行情 Actor。

CTP 连接采用懒创建模式：

- CTP MD 行情连接：首次有 Client 订阅行情时创建，全局共享。
- CTP Trade 交易连接：Client 发送 `AccountLogin` 后，按 `account_id` 创建或复用对应的 `TradeSessionActor`。
- 同一账户的所有交易命令进入同一个 `TradeSessionActor`，由 actor mailbox 串行处理。
- 多个 Client 订阅同一合约时，Server 只向 CTP 发起一次真实订阅，然后把 tick 广播给所有订阅 Client。

## 4. 使用说明

### 4.1 环境准备

运行前需要具备：

1. Linux 环境。
2. Rust toolchain 和 Cargo。
3. SimNow 账号。
4. CTP 动态库文件，目录由 `config/server.toml` 中的 `dynlib_dir` 指定。
5. `.env` 中填写 CTP 前置地址、BrokerID、AppID、AuthCode、Server MD 账号、Client 交易账号。

当前内部 TCP 端口：

| 端口 | 用途 |
|---|---|
| `8000` | Client 到 Server 的交易连接，处理登录、下单、撤单、查询、登出。 |
| `8001` | Client 到 Server 的行情连接，处理行情注册、订阅、退订和 tick 推送。 |

### 4.2 Server 配置

`config/server.toml` 负责监听地址、行情前置、动态库目录和 Client 权限白名单：

```toml
listen_addr = "0.0.0.0:8000"
market_data_listen_addr = "0.0.0.0:8001"
md_front = "tcp://182.254.243.31:30011"
dynlib_dir = "crates/server/src/adapter/exchange/ctp"

[[clients]]
client_id = "CY"
accounts = ["268809"]
permission = "Full"

[[clients]]
client_id = "HB"
accounts = ["268508"]
permission = "QueryOnly"
```

权限说明：

| 权限 | 含义 |
|---|---|
| `Full` | 可查询、可交易、可登出。 |
| `QueryOnly` | 只能执行资金、持仓、委托、成交查询。 |
| `TradeOnly` | 可执行交易类写操作，不开放查询能力。 |

`.env` 保存敏感信息和运行变量，提交或展示时应隐藏真实账号密码：

```bash
CTP_MD_FRONT=tcp://182.254.243.31:30011
CTP_TRADE_FRONT=tcp://182.254.243.31:30001

CTP_BROKER_ID=9999
CTP_APP_ID=simnow_client_test
CTP_AUTH_CODE=0000000000000000

CTP_SERVER_USER_ID=<server_md_user_id>
CTP_SERVER_PASSWORD=<server_md_password>

CTP_CLIENT_ID=CY
CTP_CLIENT_USER_ID=<trade_user_id>
CTP_CLIENT_PASSWORD=<trade_password>

CTP_INSTRUMENTS=rb2610,cu2608
RUST_LOG=info,ctp_server=debug,ctp_common=debug
```

```bash
cd /home/aaa/quantSystem/ctp_system
```

只编译 Client：

```bash
cargo build -p ctp-client
```

### 4.4 启动 Server

```bash
cd /home/aaa/quantSystem/ctp_system
set -a && source .env && set +a
./target/debug/ctp-server
```

启动成功后应看到类似日志：

```text
ctp-server starting trading_listen=0.0.0.0:8000 market_data_listen=0.0.0.0:8001
internal tcp listening addr=0.0.0.0:8000 kind=Trading
internal tcp listening addr=0.0.0.0:8001 kind=MarketData
```

### 4.5 启动单个 Client

```bash
cd /home/aaa/quantSystem/ctp_system
set -a && source .env && set +a
export CTP_CLIENT_ID=CY
export CTP_SERVER_TRADING_ADDR=127.0.0.1:8000
export CTP_SERVER_MARKET_DATA_ADDR=127.0.0.1:8001
./target/debug/ctp-client
```

Client demo 自动执行：

1. 连接 Server trading / market-data 两个端口。
2. 发送 `Hello` / `MarketDataHello` 完成服务端注册。
3. 加载 `CTP_CLIENT_ID` 对应的交易账号并发送 `AccountLogin`。
4. 订阅 `CTP_INSTRUMENTS` 指定的行情。
5. 登录成功后发送资金、持仓、委托、成交四类查询。
6. 收到 tick 后构造一笔被动限价单。
7. 收到可撤委托状态后发送撤单。
8. 撤单完成后保持订阅和连接 20 秒，方便录屏展示多 Client 同时在线。
9. 20 秒后退订行情、登出账户、退出。

### 4.6 多 Client 演示

终端 A 启动 `CY`：

```bash
cd /home/aaa/quantSystem/ctp_system
set -a && source .env && set +a

export CTP_SERVER_TRADING_ADDR=127.0.0.1:8000

./target/debug/ctp-client
```

终端 B 启动 `HB`：

```bash
cd /home/aaa/quantSystem/ctp_system
set -a && source .env && set +a
export CTP_CLIENT_ID=HB
export CTP_SERVER_TRADING_ADDR=127.0.0.1:8000
export CTP_SERVER_MARKET_DATA_ADDR=127.0.0.1:8001
./target/debug/ctp-client
```

预期现象：

- Server 日志出现两个不同 `client_id` 的注册。
- 两个 Client 可以同时连接同一个 Server。
- 行情订阅会被 Server 聚合，同一合约只在首次订阅时向 CTP 发起真实订阅。
- `CY` 是 `Full` 权限，可完成查询、下单、撤单、登出。
- `HB` 是 `QueryOnly` 权限，只应允许查询；交易类指令会被 Server 拒绝。

## 5. 测试方案

### 5.1 基础联通测试

| 用例 | 步骤 | 预期结果 |
|---|---|---|
| Server 启动 | 加载 `.env` 后运行 `ctp-server` | 监听 `8000` 和 `8001`，无 panic。 |
| Client 注册 | 运行 `ctp-client` | 收到 `HelloAck` 和 `MarketDataHello` 对应的 `HelloAck`。 |
| 行情订阅 | 设置 `CTP_INSTRUMENTS` 并运行 Client | Server 发起订阅，交易时段内 Client 打印 `MarketDataTick`。 |

### 5.2 交易流程测试

| 用例 | 步骤 | 预期结果 |
|---|---|---|
| 账户登录 | Client 发送 `AccountLogin` | Server 完成 CTP authenticate / login / settlement confirm，Client 收到 `LoggedIn`。 |
| 四类查询 | 登录后自动查询资金、持仓、委托、成交 | Client 打印四类 `Query*Result`。 |
| 下单 | Client 构造被动限价单并发送 `PlaceOrder` | Server 记录 `req_order_insert`，Client 收到 `OrderUpdate`。 |
| 撤单 | Client 对可撤订单发送 `CancelOrder` | Server 记录 `req_order_action`，Client 收到 `OrderUpdate(Cancelled)`。 |
| 登出 | 流程完成后发送 `AccountLogout` | Server 释放 Trade 会话，Client 收到 `Disconnected` 后退出。 |

### 5.3 权限测试

| 用例 | 步骤 | 预期结果 |
|---|---|---|
| `Full` 权限 | 使用 `CTP_CLIENT_ID=CY` | 可查询、下单、撤单、登出。 |
| `QueryOnly` 权限 | 使用 `CTP_CLIENT_ID=HB` | 查询成功；下单、撤单、登出等写操作被拒绝。 |
| 未授权账户 | Client 请求 TOML 未授权的 `account_id` | Server 返回 `permission denied` 错误。 |

### 5.4 断线恢复测试

| 用例 | 步骤 | 预期结果 |
|---|---|---|
| Client TCP 重连 | Client 运行期间重启 Server | Client TCP 自动重连，并重放 `Hello`、登录和订阅消息。 |
| MD 重订阅 | CTP MD 登录恢复后 | Server 根据 `market_data_subscriptions` 自动重新订阅仍有 Client 订阅的合约。 |

注意：行情是否持续推送取决于交易时段。日盘收盘后可能只收到 `15:00:01` 附近的最后 tick，订阅仍然有效但不会有新增行情。

## 6. 代码说明

### 6.1 Workspace 模块

| Crate | 职责 |
|---|---|
| `ctp-model` | 领域模型：强类型 ID、账户、订单、成交、持仓、枚举和回报结构。 |
| `ctp-common` | 公共基础设施：Client/Server 协议、TCP 客户端、CTP MD 客户端封装、日志初始化。 |
| `ctp-server` | 服务端：内部 TCP gateway、权限校验、Actor、CTP Trade/MD adapter。 |
| `ctp-client` | 客户端：交易 API、行情 API、凭证加载、端到端 demo。 |

### 6.2 协议层

`crates/common/src/protocol.rs` 定义 Client 与 Server 的统一消息协议：

- `Envelope`：外层消息，包含 `request_id`、`client_id`、时间戳和 `payload`。
- `Message`：业务消息枚举，包括 `Hello`、`AccountLogin`、`PlaceOrder`、`CancelOrder`、`Query*`、`MarketDataTick`、`Error` 等。
- 网络帧格式为 `u32 BE length + JSON payload`，方便跨语言实现 Client。

### 6.3 Server 端

`crates/server/src/main.rs`：

- 加载 `ServerConfig`。
- 初始化共享 `ServerActorState`。
- 启动 `InternalGateway`，同时监听交易端口和行情端口。

`crates/server/src/adapter/internal/tcp_server.rs`：

- 接受 Client TCP 连接。
- 将长度前缀帧解码为 `Envelope`。
- 根据 endpoint 类型和 `Message` 类型路由到 `ServerActorState`。
- 为每条连接维护 `outbound_tx`，用于 Server 向该 Client 推送消息。

`crates/server/src/session.rs`：

- 管理 `ClientId`、`AccountId` 和权限白名单关系。
- `client` 表示接入 Server 的终端身份。
- `account` 表示真实 CTP 交易账户。
- 在登录、查询、交易、登出入口执行读写权限校验。

`crates/server/src/actor.rs`：

- `ServerActorState` 保存全局运行状态：Client 连接、MarketData 订阅表、Trade session actors。
- `TradeSessionActor` 按账户懒创建，同一账户所有命令串行进入该 Actor。
- `MarketDataSessionActor` 全局共享，首次有真实订阅时懒创建。
- 行情订阅按合约聚合，只有从 0 个订阅者变为 1 个订阅者时才向 CTP 订阅；最后一个订阅者离开时才向 CTP 退订。

`crates/server/src/trade_actor_ops.rs`：

- 实现已登录和未登录状态下的交易命令处理。
- 包括登录、下单、撤单、资金查询、持仓查询、委托查询、成交查询和登出。
- 撤单逻辑会处理 CTP 异步响应顺序。

`crates/server/src/adapter/exchange/trade.rs`：

- 封装 CTP Trade API 生命周期。
- 处理 authenticate、login、settlement confirm、order insert、order action、query、logout。
- SPI 回调被转换为内部 `TradeSessionEvent`。

`crates/server/src/adapter/exchange/md.rs`：

- 封装 CTP MD API 生命周期。
- 处理连接、登录、订阅、退订和行情回调。
- 行情回调转换为内部 tick 后由 Server fanout 到订阅 Client。

### 6.4 Client 端

`crates/client/src/connection.rs`：

- 封装通用 TCP 连接。
- 将收到的 JSON payload 解码为 `Envelope`。
- 支持登记重连后需要重放的业务消息。

`crates/client/src/trading.rs`：

- 暴露交易 API：`connect`、`login_account`、`place_order`、`cancel_order`、`query_account`、`query_position`、`query_orders`、`query_trades`、`logout_account`。
- TCP 重连后自动重放 `Hello` 和最近一次登录凭证。

`crates/client/src/market_data.rs`：

- 暴露行情 API：`connect`、`subscribe`、`unsubscribe`。
- 本地维护订阅集合，TCP 重连后自动重放 `MarketDataHello` 和当前全部订阅。
- 使用 `BTreeMap` 保存订阅，是为了让重连 replay 和日志顺序稳定。

`crates/client/src/credentials.rs`：

- 根据当前进程环境变量加载一个 Client 实例的交易账号。
- `CTP_CLIENT_ID` 表示 Server 识别的逻辑 Client 身份。
- `CTP_CLIENT_USER_ID` 和 `CTP_CLIENT_PASSWORD` 表示本次启动要登录的 CTP 交易账号。

`crates/client/src/main.rs`：

- 端到端演示程序。
- 自动完成连接、注册、登录、订阅、四类查询、下单、撤单、保持连接 20 秒、退订、登出。
- 适合现场演示和录屏。

## 7. 设计分析

### 7.1 Client 和 Account 的区别

`ClientId` 是接入 Server 的终端身份，例如 `CY`、`HB`。它用于连接管理、权限校验和消息路由。

`AccountId` 是真实 CTP 交易账户，例如 `268809`。它用于 CTP Trade 登录、资金、持仓、委托和成交。

这种拆分允许一个 Server 同时接入多个终端，并通过白名单控制不同终端可访问哪些账户、具备何种权限。

### 7.2 Trade 为什么按账户建 Actor

CTP Trade 会话天然围绕交易账户展开。按账户创建 `TradeSessionActor` 有几个好处：

1. 同一账户的登录状态、front/session id、订单上下文集中维护。
2. 同一账户的命令通过 actor mailbox 串行处理，避免并发修改状态。
3. 不同账户互不阻塞，可以并行运行。

### 7.3 MD 为什么全局共享

行情与交易账户无强绑定，多个 Client 订阅同一合约时没有必要重复连接 CTP MD 或重复订阅。因此 Server 维护一个共享 `MarketDataSessionActor`：

1. 减少 CTP 行情前置连接数。
2. 对同一合约做订阅去重。
3. 收到 tick 后按订阅表广播给所有 Client。

### 7.4 TCP + JSON Envelope

TCP 长连接适合交易系统内部通信，长度前缀帧避免粘包拆包问题。JSON `Envelope` 便于调试和跨语言扩展，现场演示时也容易从日志理解消息内容。

## 8. 已知限制与后续优化

| 项目 | 说明 |
|---|---|
| Trade 断线补偿 | 已有基础事件处理，但 Trade front 断线后的完整重登、订单/成交补查补偿仍可继续增强。 |
| 查询流控 | CTP 查询接口存在频率限制，当前 demo 串行发送查询；后续可加统一 query throttle。 |
| 历史委托归属 | 查询到历史委托时，不一定都存在本进程 `order_registry` 中，归属 Client 的精确恢复可继续增强。 |
| 权限粒度 | 当前粒度为 `client_id + account_id + read/write`，未细分到合约、方向、额度。 |
| 汇总面板 | 暂未实现账户汇总 CLI 或 Web 面板。 |
